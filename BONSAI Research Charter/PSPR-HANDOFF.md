# BONSAI PSPR Handoff

Version: 0.1  
Status: Inputs and PSPR v0.1 approved; execution not authorized

Approved PSPR: [BONSAI-CANONICAL-PLAN-SEQUENTIAL-PROMPT-ROSTER.md](./BONSAI-CANONICAL-PLAN-SEQUENTIAL-PROMPT-ROSTER.md)

## 1. Purpose

This document provides the boundary between the research charter and the future granular Canonical Plan / Sequential Prompt Roster Protocol document.

The PSPR must convert the charter into an executable, auditable, cross-platform implementation sequence. It must preserve the charter's scientific neutrality, evidence standards, and exclusions.

## 2. Settled initiative

Build a comprehensive and robust measurement and resource-governance layer that determines whether an OaK-style discovery cycle is producing useful state and temporal abstractions under controlled resources.

## 3. Settled name

**BONSAI** — **Benchmark for Online, Nonstationary, Single-pass Agent Intelligence**.

## 4. Settled target platforms

- Windows
- macOS
- Linux

All three are first-class targets. CI evidence and physical-host acceptance evidence must be distinguished.

## 5. Settled governance

- The charter is authoritative for scientific intent.
- The PSPR will be authoritative for implementation order and gates after approval.
- Drafting the PSPR is not authorization to execute it.
- Execution requires the user to say `run it STS` or explicitly approve named phases or prompts.
- One focused prompt should normally map to one focused commit, with a development log recording verification and commit identity.
- No prompt is complete until its prescribed checks pass.
- Scope changes, superseding decisions, and parked work must be recorded honestly.

## 6. Mandatory PSPR workstreams

The granular roster must cover, in dependency order:

1. Repository and documentation governance.
2. Cross-platform runtime and packaging selection.
3. Canonical terminology, identifiers, units, and schema versioning.
4. Experiment manifest and result-bundle formats.
5. Event ingestion and ordering semantics.
6. Cognitive-artifact lifecycle and lineage graph.
7. Agent/observer data isolation and replay classification.
8. Platform-neutral resource-accounting interfaces.
9. Windows resource-measurement backends.
10. macOS resource-measurement backends.
11. Linux resource-measurement backends.
12. Energy evidence tiers and calibration metadata.
13. External budget governor and fail-closed behavior.
14. Scheduler instrumentation for dense and event-driven candidates.
15. Metric engine for behavior and continual learning.
16. Feature, subproblem, option, model, and planning metrics.
17. Discovery-cycle utility-credit metrics.
18. Reference STOMP/OaK-style linear or tabular agent adapter.
19. Primitive, fixed-feature, no-planning, and replay comparator adapters.
20. Reference scenario families and deterministic fixtures.
21. Ablation orchestration and matched-budget comparison.
22. Claim-ladder adjudication engine.
23. Reporting and visualization.
24. Instrumentation-overhead measurement.
25. Cross-platform equivalence and reproducibility gates.
26. Long-duration acceptance runs.
27. Security, integrity, and denial-of-service controls.
28. Documentation, examples, release packaging, and final evidence audit.

These workstreams are headings for later decomposition, not a sufficient prompt roster.

## 7. Required plan artifacts

The PSPR must define or create:

- governance header and source-of-truth hierarchy;
- architecture decision records for settled stack choices;
- reuse ledger;
- stable prompt identifiers;
- prompt dependency graph;
- independently approvable milestones;
- per-prompt objective, files, exclusions, and acceptance gate;
- test matrix by platform and evidence level;
- development log format;
- verification log format;
- risk and blocker register;
- parked-scope ledger;
- claim-to-evidence matrix;
- completion and publication criteria.

## 8. Questions requiring decisions during PSPR drafting

### Architecture

- Primary language and runtime.
- Single process, plugin host, or process-isolated adapters.
- Embedded database, append-only files, or combined telemetry storage.
- Schema and API evolution strategy.
- Local dashboard, static report generation, or both.

### Measurement

- Required clock resolution and overhead ceiling.
- Canonical compute proxies when hardware counters are unavailable.
- Supported accelerator vendors in the first milestone.
- Minimum energy evidence required for specific claims.
- Statistical seed counts and confidence requirements.

### Reference science

- First environments for each scenario family.
- First reference agent and control implementations.
- Exact resource envelopes for smoke, conformance, and acceptance tiers.
- Formal definition of a consequential planning backup.
- Initial utility estimators and acceptable substitutes.

### Distribution

- License.
- Public versus private repository.
- Binary, source, container, or combined release formats.
- Offline dependency and reproducible-build expectations.

## 9. PSPR acceptance gate

The PSPR will be ready for execution review only when:

1. Every charter requirement maps to one or more stable prompts and verification gates.
2. Platform work is explicit rather than deferred to generic portability language.
3. Measurement limitations and energy tiers are reflected in the roster.
4. Strict experiential and comparator tracks are structurally separated.
5. Reference implementations cannot be mistaken for unpublished Oak Lab algorithms.
6. Milestones deliver usable vertical slices, not only horizontal infrastructure.
7. Security, integrity, and long-duration evidence are scheduled before completion claims.
8. The reuse ledger and authoritative working location are settled.
9. Open decisions that would materially alter the roster have been presented for user approval.
10. The document clearly states that execution awaits explicit STS authorization.
